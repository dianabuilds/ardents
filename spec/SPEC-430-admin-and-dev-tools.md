# SPEC-430: Инструменты администрирования и разработки

**Статус:** Approved v1.0 (2026-02-02)  
**Зависимости:** SPEC-010, SPEC-140, SPEC-420  
**Назначение:** зафиксировать набор dev-инструментов, необходимых для отладки протокола.

---

## 1) Packet capture (фиксировано)

Узел **ДОЛЖЕН** уметь включать режим захвата overlay-сообщений:

* запись всех входящих/исходящих Envelope (в wire-виде) в файл `run/pcap.jsonl`
* каждая запись — объект с:
  * `dir` (`in`|`out`)
  * `ts_ms`
  * `peer_id_remote` (если известен)
* `envelope_cbor_b64` (base64)

Требования безопасности (фиксировано):

* packet capture **ДОЛЖЕН** быть `off` по умолчанию;
* файл `run/pcap.jsonl` **ДОЛЖЕН** создаваться с owner-only правами (на POSIX `0600`, на Windows — ограниченный ACL);
* при включении capture узел **ДОЛЖЕН** логировать `warn` с `event=pcap.enabled`.

Примечание по путям:

* `run/` — это runtime-директория узла (см. `spec/SPEC-100-network-manager.md` и `docs/TECH-050-operations-and-service.md`).

---

## 2) Replay (фиксировано)

Должна существовать утилита replay, которая:

* читает `run/pcap.jsonl`
* воспроизводит отправки сообщений с сохранением относительных задержек (или режим “без задержек”).

Replay в v1 **ДОЛЖЕН** быть “sandbox-only” по умолчанию:

* разрешены только loopback соединения и локальный симулятор;
* воспроизведение в реальную сеть требует явного флага `--allow-network` (который по умолчанию выключен).

---

## 3) Simulator (фиксировано)

Должен существовать симулятор, который:

* поднимает N виртуальных peer в одном процессе
* генерирует трафик (task/fetch)
* измеряет latency, drop rate, pow reject rate

Simulator **ДОЛЖЕН** соблюдать все лимиты и требования PoW/TTL/dedup как реальный peer (SPEC-002/100/140), иначе результаты симуляции считаются недостоверными.

---

## 4) CLI-утилиты (фиксировано)

ДОЛЖНЫ существовать минимальные CLI-утилиты для локальной проверки протокола:

* `node` — просмотр Content Nodes:
  * команда `node get --target <ufa> [--decrypt] [--history-depth N]`;
  * делает `node.fetch.v1` через runtime и выводит JSON с метаданными/links;
  * при `--decrypt` пытается расшифровать `enc.node.v1` (если есть ключи).
* `webclient` — ручной запрос к сервису `web.request.v1`:
  * команда `webclient request --addr <host:port> --service-id <service_id> [--peer-id] ...`;
  * отправляет `task.request.v1`, печатает ACK и `result_node_id`;
  * при `--fetch-result` выполняет `node.fetch.v1` и печатает ответ (status/headers/body).
