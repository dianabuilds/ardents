# SPEC-425: Обновления и миграции локальных данных

**Статус:** Draft v1.0 (2026-02-04)  
**Зависимости:** SPEC-010, SPEC-420, TECH-050  
**Назначение:** зафиксировать требования к версиям локальных данных, процедурам обновления и миграциям.

---

## 1) Область действия (scope)

В рамках v1 описываются **локальные** обновления и миграции данных узла:

* `config/` и `data/` под `ARDENTS_HOME`/XDG.
* Проверка совместимости при старте.
* Команда миграции и журнал миграций.

**НЕ ВХОДИТ В v1:**

* Автоматические миграции при старте (без явной команды).
* Сетевые/удалённые миграции.
* Миграции логов/pcap/`run/`.
* Автоматический rollback.

---

## 2) Термины и определения

* **Schema Version** — версия формата локальных данных узла (целое число).
* **Migration** — процедура перевода локальных данных с версии N на версию N+1.
* **Supported Version** — минимальная и максимальная версии, которые поддерживает бинарник.

---

## 3) Версия локальных данных

### 3.1 Файл версии

Узел **ДОЛЖЕН** хранить версию данных в `data/version.json`.

Формат (JSON, UTF-8):

```
{
  "schema_version": 1,
  "created_at_ms": 1700000000000,
  "updated_at_ms": 1700000000000
}
```

Правила:

* `schema_version` **ДОЛЖЕН** быть `int64` и начинаться с 1.
* `created_at_ms` и `updated_at_ms` **ДОЛЖНЫ** быть UTC `int64` в миллисекундах.
* `updated_at_ms` **ДОЛЖЕН** обновляться после успешной миграции.

### 3.2 Инициализация

Команда `peer init` **ДОЛЖНА** создавать `data/version.json` со значением `schema_version=1`.

---

## 4) Команда миграции

### 4.1 Общие требования

Миграции **ДОЛЖНЫ** выполняться только явной командой:

* `peer migrate`

Команда **НЕ ДОЛЖНА** выполнять сетевых операций.

### 4.2 Идемпотентность

`peer migrate` **ДОЛЖНА** быть идемпотентной: повторный запуск не изменяет данные, если `schema_version` уже соответствует максимальной поддерживаемой версии.

---

## 5) Backup перед миграцией

Перед выполнением миграций `peer migrate` **ДОЛЖНА** создать локальный backup, если не указан флаг `--no-backup`.

Требования к backup:

* Путь: `run/backup-<ts_ms>/`.
* Структура: `run/backup-<ts_ms>/config/` и `run/backup-<ts_ms>/data/`.
* Backup **ДОЛЖЕН** быть полным (копии всех файлов и подкаталогов).

---

## 6) Журнал миграций

`peer migrate` **ДОЛЖНА** записывать журнал миграций в `run/migrations.jsonl` (JSON Lines).

Каждая запись **ДОЛЖНА** содержать поля:

* `ts_ms` (int64)
* `from_version` (int64)
* `to_version` (int64)
* `status` (`started` | `applied` | `failed`)
* `error_code` (string, optional)

---

## 7) Поведение при старте

Команда `peer start` **ДОЛЖНА** проверять `data/version.json`:

* если `schema_version` ниже минимальной поддерживаемой — **ОТКАЗАТЬ** с `ERR_MIGRATION_REQUIRED`.
* если `schema_version` выше максимальной поддерживаемой — **ОТКАЗАТЬ** с `ERR_MIGRATION_UNSUPPORTED`.

Запуск **НЕ ДОЛЖЕН** продолжаться в состоянии несовместимости.

---

## 8) Ошибки (error codes)

Минимальные стабильные коды:

* `ERR_MIGRATION_REQUIRED`
* `ERR_MIGRATION_UNSUPPORTED`
* `ERR_MIGRATION_FAILED`
* `ERR_MIGRATION_BACKUP_FAILED`

---

## 9) Безопасность

* Миграции **НЕ ДОЛЖНЫ** отправлять данные по сети.
* Миграции **НЕ ДОЛЖНЫ** изменять ключи, токены или секреты.
* Логи миграций **НЕ ДОЛЖНЫ** содержать секреты.

---

## 10) User stories

1. **Оператор обновляет бинарник** и запускает `peer migrate`, после чего `peer start` проходит без ошибок.
2. **Оператор запускает `peer start` без миграции** и получает явную ошибку `ERR_MIGRATION_REQUIRED`.
3. **Оператор повторно запускает `peer migrate`** — команда завершается без изменений (идемпотентность).

---

## 11) Edge cases

1. `data/version.json` отсутствует → `peer start` возвращает `ERR_MIGRATION_REQUIRED`.
2. Миграция прервана/упала → в журнале фиксируется `status=failed`, данные не считаются обновлёнными.
