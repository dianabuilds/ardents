# SPEC-126: Автоматическое обнаружение и резолвинг ресурсов

**Статус:** Draft (2026-02-04)  
**Зависимости:** SPEC-001, SPEC-010, SPEC-120, SPEC-200, SPEC-210, SPEC-300, SPEC-415, SPEC-510, SPEC-420  
**Назначение:** описать автоматическую систему обнаружения ресурсов (services/nodes) и резолвинга адресов без привязки к конкретным профилям транспорта, при сохранении локальной trust‑политики.

---

## 1) Scope / Out-of-scope

### Scope

* Автоматическое обнаружение ресурсов при отсутствии прямого подключения.
* Источники кандидатов, порядок приоритета и детерминированный выбор.
* Кэширование, TTL и обновление результатов.
* Наблюдаемость и минимальный набор ошибок.

### Out-of-scope

* Глобальный «реестр имён» как единственный источник истины.
* Автоматический поиск «по имени» в глобальной сети без локальной trust‑политики (см. SPEC-120).
* Транспортные детали/туннели.

---

## 2) Термины

* **Resource** — объект, который нужно найти: сервис или контент‑узел.
* **Descriptor** — объявление сервиса/возможностей (service descriptor).
* **Provider** — узел, объявивший хранение конкретного `node_id`.
* **Directory** — внешний каталог кандидатов, заданный локальной конфигурацией и подчинённый локальной trust‑политике.
* **Local trust** — доверие, определяемое локальным address book (SPEC-120).

---

## 3) Требования (нормативные)

1) Система **ДОЛЖНА** поддерживать автоматическое обнаружение без ручного ввода адресов узлов.  
2) Система **ДОЛЖНА** сохранять локальный trust boundary: все источники данных проверяются по trust‑политике.  
3) Результат **ДОЛЖЕН** быть детерминирован при одинаковых входных данных.  
4) Система **НЕ ДОЛЖНА** полагаться на единственный источник истины.  
5) Система **ДОЛЖНА** иметь кэш с TTL и механизм обновления (refresh).

---

## 4) Источники данных и приоритет

Система **ДОЛЖНА** собирать кандидатов из следующих источников (в указанном порядке приоритета):

1) **Local address book (trusted).**  
   Записи с `trust=trusted` имеют высший приоритет (SPEC-120).

2) **Session/known peers.**  
   Узлы, с которыми недавно был успешный обмен (локальный кэш с TTL).

3) **Provider records.**  
   Объявления хранения `node_id` с валидным TTL.

4) **Directory results.**  
   Результаты запроса к каталогу, который явно добавлен в локальную конфигурацию и имеет заданные правила доверия.  
   Если каталоги не настроены локально, этот источник считается отключённым и не используется.

Если источник недоступен или даёт пустой результат, система **ДОЛЖНА** перейти к следующему.

---

## 5) Алгоритм обнаружения (обобщённый)

### 5.1 Поиск сервиса

1) Система строит запрос по `service_name`/capabilities (SPEC-200/SPEC-210).  
2) Получает список кандидатов (service descriptors).  
3) Отбрасывает невалидные/неtrusted дескрипторы.  
4) Сортирует кандидатов по правилам приоритета и метрикам доступности.  
5) Возвращает top‑N кандидатов для соединения.

### 5.2 Поиск контент‑узла (`node_id`)

1) Проверить локальный кэш/хранилище.  
2) Использовать session peer (если известен).  
3) Использовать provider records.  
4) Использовать directory, только если он явно включён локальной конфигурацией.  
5) Если не найдено — вернуть `ERR_RESOURCE_NOT_FOUND`.

---

## 6) Кэширование и TTL

* Любые результаты обнаружения **ДОЛЖНЫ** кэшироваться с TTL.  
* Просроченные записи **НЕ ДОЛЖНЫ** использоваться для соединения.  
* При успешном соединении записи **ДОЛЖНЫ** обновляться (refresh).

---

## 7) Безопасность и доверие

* Система **ДОЛЖНА** проверять подписи/идентификаторы источников согласно SPEC-001/SPEC-120.  
* Результаты из untrusted источников **НЕ ДОЛЖНЫ** иметь приоритет над trusted.  
* Должна быть защита от подмены/poisoning через подписи и локальные политики.

---

## 8) Ошибки (минимум)

* `ERR_RESOURCE_NOT_FOUND` — не найден ни один кандидат.  
* `ERR_DIRECTORY_UNAVAILABLE` — каталог недоступен (если включён локальной конфигурацией).  
* `ERR_DESCRIPTOR_INVALID` — descriptor невалиден/неtrusted.  
* `ERR_PROVIDER_UNAVAILABLE` — нет доступных provider‑адресов.

---

## 9) Наблюдаемость

События (минимум):

* `discovery.query.started`
* `discovery.query.finished`
* `discovery.candidates.selected`
* `discovery.source.unavailable`

Логи **ДОЛЖНЫ** содержать источник, количество кандидатов и выбранные id.
