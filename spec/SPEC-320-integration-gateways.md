﻿# SPEC-320: Интеграционные шлюзы (Integration Gateways)

**Статус:** Approved v1.0 (2026-02-02)  
**Зависимости:** SPEC-010, SPEC-140, SPEC-300  
**Назначение:** определить, как внешние системы интегрируются с overlay-сетью через локальный IPC-адаптер, не нарушая trust boundaries.

---

## 1) Принцип (фиксировано)

Интеграция выполняется через **локальный IPC-адаптер**:

* внешний вычислитель/сайт подключается к runtime по IPC;
* регистрирует `service_name` и `job_type` для Tasks;
* получает `task.request.v1` и возвращает `task.result.v1` (через Node bytes).

Публичный/внешний HTTP gateway **ЗАПРЕЩЁН**. Если требуется HTTP-интерфейс, он реализуется **только** как локальный адаптер поверх IPC и **ДОЛЖЕН**:

* слушать **только** loopback;
* требовать тот же token (Authorization: Bearer);
* не принимать подключения с внешних интерфейсов;
* не расширять модель доверия (любой HTTP-вход трактуется как локальный IPC-клиент).

---

## 2) Локальный IPC (фиксировано)

Peer runtime **ДОЛЖЕН** предоставлять локальный IPC‑интерфейс `Integration.v1`:

* на Unix: Unix domain socket `run/peer.sock`
* на Windows: Named Pipe `\\\\.\\pipe\\ardents-peer`

Интерфейс **ДОЛЖЕН** требовать аутентификацию локальным токеном:

* токен хранится в `run/peer.token` (случайные 32 bytes, base64) и читается только владельцем процесса.

Требования к безопасности файла токена (фиксировано):

* токен-файл **ДОЛЖЕН** быть доступен на чтение только владельцу процесса (owner-only).
  * на POSIX: режим **ДОЛЖЕН** быть `0600`;
  * на Windows: ACL **ДОЛЖЕН** разрешать чтение только текущему пользователю/сервисному аккаунту.
* директория, содержащая `run/peer.token` и `run/peer.sock`, **ДОЛЖНА** быть owner-only.
  * на POSIX: режим директории **ДОЛЖЕН** быть `0700`;
  * на Windows: правило не применяется (ACL задаётся на Named Pipe и token-файл).
* если обеспечить owner-only права не удалось, peer **ДОЛЖЕН** отказать в запуске IPC с ошибкой `ERR_GATEWAY_UNAUTHORIZED`.

Требования к безопасности IPC на Windows (фиксировано):

* Named Pipe **ДОЛЖЕН** создаваться с owner-only ACL (доступ только текущему пользователю/сервисному аккаунту).
* если установить owner-only ACL для Named Pipe не удалось, peer **ДОЛЖЕН** отказать в запуске IPC с ошибкой `ERR_GATEWAY_UNAUTHORIZED`.

Rotation (фиксировано):

* токен **ДОЛЖЕН** ротироваться при каждом старте peer runtime (новый случайный токен, старый становится невалидным).
* при старте IPC **ДОЛЖЕН** логировать `warn`, что token является “секретом”.

## 3) Протокол `Integration.v1` (фиксировано)

Формат сообщений — JSON objects (stream, одна JSON‑структура на сообщение).

### 3.1 Аутентификация

Первое сообщение клиента **ДОЛЖНО** быть:

```
{ "type": "auth", "token": "<base64>" }
```

### 3.2 Регистрация сервиса

После auth клиент **ДОЛЖЕН** отправить:

```
{ "type": "register", "service_name": "web.request.v1", "job_types": ["web.request.v1"] }
```

Требования:

* `service_name` **ДОЛЖЕН** быть валидным service_name (SPEC‑001/300).
* `job_types` **ДОЛЖЕН** содержать как минимум один job_type; текущая реализация требует, чтобы `job_types[0] == service_name`.

Ответ:

```
{ "type": "registered", "service_id": "<service_id>" }
```

### 3.3 Доставка задач

Runtime отправляет:

```
{ "type": "task", "task_id": "...", "client_request_id": "...", "job_type": "web.request.v1", "input": { ... }, "ts_ms": 123 }
```

Клиент отвечает:

```
{ "type": "result", "task_id": "...", "result_node": "<base64-cbor-bytes>" }
```

Ошибки:

```
{ "type": "result", "task_id": "...", "error_code": "ERR_INTEGRATION_FAILED", "error_message": "..." }
```

---

## 4) Ошибки (минимум)

* `ERR_GATEWAY_UNAUTHORIZED`
* `ERR_GATEWAY_FORBIDDEN_BIND`
* `ERR_GATEWAY_BAD_REQUEST`
