# SPEC-130: Routing и Relays (v1 direct mode)

**Статус:** Approved v1.0 (2026-02-02)  
**Зависимости:** SPEC-001, SPEC-010, SPEC-110, SPEC-140  
**Назначение:** определить доставку сообщений в профиле v1: прямую и через relay-хопы. Privacy-first (туннели/garlic) определён отдельно в SPEC-520 и заменяет этот механизм.

---

## 1) Scope v1 (фиксировано)

Routing v1 поддерживает:

* **Direct delivery**: отправка напрямую на `peer_id` назначения (если известен адрес).
* **Relay delivery**: доставка через фиксированное число relay-хопов `N` (N=1 или N=2).

Большее N в v1 **НЕ ДОПУСКАЕТСЯ**.

Ordering сообщений в routing v1 **НЕ ГАРАНТИРУЕТСЯ**.

---

## 2) Требование конфиденциальности маршрута (фиксировано)

Промежуточный relay **НЕ ДОЛЖЕН** знать конечного адресата (`to`) и содержимое payload.

Для этого используется “луковичная” упаковка relay-пакетов:

* каждый хоп видит только `next_peer_id` и зашифрованный `inner_packet`;
* `inner_packet` шифруется под транспортный публичный ключ следующего хопа.

---

## 3) RelayPacket.v1 (wire-формат)

RelayPacket — CBOR map:

* `v` = 1
* `next_peer_id` (string)
* `ttl_ms` (int64)
* `inner` (bytes) — зашифрованный пакет для следующего хопа

Шифрование `inner`:

* используется sealed-box к транспортному ключу получателя (peer transport key);
* симметрический алгоритм: XChaCha20-Poly1305 (см. SPEC-010).

Ключи:

* транспортный Ed25519 ключ **ДОЛЖЕН** быть детерминированно преобразуем в X25519 (Montgomery) для целей sealed-box.

---

## 4) Поведение relay-хопа

Получив RelayPacket, relay **ДОЛЖЕН**:

1. проверить `ttl_ms` (если истёк — отклонить);
2. расшифровать `inner` своим транспортным приватным ключом;
3. получить следующий объект (RelayPacket или финальный Envelope из SPEC-140);
4. отправить на `next_peer_id`.

Relay **НЕ ДОЛЖЕН**:

* модифицировать `inner`;
* пытаться интерпретировать финальный payload (кроме минимальной маршрутизации следующего хопа).

---

## 5) Ошибки (минимум)

* `ERR_RELAY_TTL_EXPIRED`
* `ERR_RELAY_DECRYPT_FAILED`
* `ERR_RELAY_NEXT_HOP_UNREACHABLE`

---

## 6) ACK и маршрутизация (фиксировано)

* ACK является обычным Envelope типа `ack.v1` по SPEC-140.
* Получатель **ДОЛЖЕН** адресовать ACK на `from.peer_id` исходного сообщения.
* Доставка ACK выполняется детерминированно:
  * если `from.peer_id` достижим напрямую — ACK **ДОЛЖЕН** быть отправлен напрямую;
  * иначе ACK **ДОЛЖЕН** быть отправлен через relay (N=1..2) по текущей политике routing узла.
