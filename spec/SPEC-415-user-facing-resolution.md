# SPEC-415: Пользовательская адресация и разрешение источника (v1)

**Статус:** Draft (2026-02-04)  
**Зависимости:** SPEC-120, SPEC-140, SPEC-200, SPEC-210, SPEC-300, SPEC-310, SPEC-410, SPEC-510, SPEC-460  
**Назначение:** убрать необходимость ввода `peer_id` пользователем и описать, как клиент выбирает источник для получения результата/Node, а также как обрабатывается единая пользовательская строка адреса.

---

## 1) Scope / Out-of-scope

### Scope

Этот SPEC определяет пользовательский UX и алгоритм разрешения источника для:

* получения `Content Node` по `node_id` (CIDv1);
* получения результата `task.*` (когда результатом является `node_id`).
* режим клиента без явного `addr` при наличии `client.json` (SPEC-460).

SPEC **НЕ МЕНЯЕТ** wire-форматы сообщений (SPEC-140/200/210/310) и **НЕ ВВОДИТ** новые поля в протокол.

### Out-of-scope

* глобальный реестр имён;
* поиск по произвольным человекочитаемым именам в сети;
* изменение форматов идентификаторов (SPEC-001).

---

## 2) Термины

* **User-facing ID** — идентификатор, который вводит пользователь (например, `service_id`, `owner_id`, `node_id`).
* **User-facing address (UFA)** — единая строка, вводимая пользователем вместо набора идентификаторов/флагов. UFA однозначно резолвится в `target_type` + `target_id` по правилам данного SPEC.
* **Source peer** — peer, у которого клиент делает `node.fetch`.
* **Session peer** — peer, с которым клиент уже установил успешное соединение для запроса задачи.

---

## 3) Требования UX

1) Клиент **НЕ ДОЛЖЕН** требовать от пользователя явный ввод `peer_id` для получения результата задачи или `node_id`.  
2) Пользователь **ДОЛЖЕН** вводить только:
   * `service_id` или `owner_id` (для задач/интеграций), **ИЛИ**
   * `node_id` (для просмотра Node).  
3) Ввод `peer_id` **МОЖЕТ** быть доступен только как *advanced/debug* режим.  
4) Клиент **ДОЛЖЕН** принимать один текстовый параметр (UFA) вместо набора разрозненных полей/флагов, и сам выполнять резолвинг `target_type`/`target_id`.

---

## 4) Формат и разбор UFA (единая пользовательская строка)

Клиент **ДОЛЖЕН** применять правила ниже **в указанном порядке**:

1) **Alias (address book).**  
   Если UFA соответствует `alias` из локального address book (SPEC-120), клиент **ДОЛЖЕН** резолвить его в `target_type` и `target_id` записи.

2) **Node ID.**  
   Если UFA является валидным `node_id` (CIDv1, base32 lower; SPEC-001), то `target_type = node`, `target_id = node_id`.

3) **Service ID.**  
   Если UFA является валидным `service_id` (префикс `svc_`; SPEC-001), то `target_type = service`, `target_id = service_id`.

4) **Owner ID (identity).**  
   Если UFA является валидным `identity_id` (формат `did:key:...`; SPEC-001), то `target_type = service`, а `target_id` **ДОЛЖЕН** быть вычислен по правилам раздела 5 (контекст клиента и `service_name`).

5) **Иначе** клиент **ДОЛЖЕН** вернуть `ERR_UFA_UNSUPPORTED`.

UFA **НЕ ДОЛЖНА** содержать пробелы и **ДОЛЖНА** нормализоваться в lowercase перед разбором.

---

## 5) Контекст клиента и выбор `service_name`

1) Каждый клиент **ДОЛЖЕН** иметь фиксированное `service_name` по умолчанию (SPEC-001, раздел 6.2).  
2) Если UFA резолвится как `identity_id`, клиент **ДОЛЖЕН** вычислять `service_id` из `(identity_id, service_name)` по SPEC-001.  
3) Если клиент не может определить `service_name`, он **ДОЛЖЕН** вернуть `ERR_SERVICE_NAME_REQUIRED`.

Примеры обязательных значений:

* web client: `web.request.v1`
* task client: профильный `service_name` по SPEC-300/310

---

## 6) Алгоритм разрешения источника (Node Fetch)

Клиент **ДОЛЖЕН** выполнять шаги в указанном порядке:

1) **Разрешить UFA в `node_id`.**  
   Если `target_type != node`, клиент **ДОЛЖЕН** вернуть `ERR_UFA_TYPE_MISMATCH`.

2) **Локальный кэш/хранилище.**  
   Если `node_id` найден локально, вернуть его без сетевого запроса.

3) **Session peer (если известен).**  
   Если `node_id` получен как результат задачи, клиент **ДОЛЖЕН** попытаться получить Node у того же peer, который вернул `task.result`/`task.fail`.

4) **Providers / NetDB.**  
   Если session peer неизвестен или fetch неуспешен, клиент **ДОЛЖЕН** использовать список provider-узлов (SPEC-210/510) для `node_id`.

5) **Trusted peers (Address Book).**  
   При наличии trusted peer-записей (SPEC-120) клиент **ДОЛЖЕН** предпочитать их как кандидатов (при прочих равных).

Если после всех шагов Node не получен, клиент **ДОЛЖЕН** вернуть ошибку `ERR_NODE_NOT_FOUND` (SPEC-210).

---

## 7) Алгоритм разрешения источника (Task Result)

Если результатом задачи является `node_id`:

1) Клиент **ДОЛЖЕН** запомнить `session peer` (peer, с которого пришел `task.result`).
2) Для получения результата **ДОЛЖЕН** применяться алгоритм из раздела 6.

---

## 8) Безопасность и доверие

* Клиент **ДОЛЖЕН** верифицировать CID и подпись Node (SPEC-200).  
* Клиент **ДОЛЖЕН** обрабатывать ошибки `ERR_NODE_CID_MISMATCH` и `ERR_NODE_SIG_INVALID` как критические.  
* Использование trusted peers **НЕ ДОЛЖНО** снижать требования по проверке CID/подписей.

---

## 9) Ошибки (минимум)

Клиент **ДОЛЖЕН** использовать следующие ошибки:

* `ERR_NODE_NOT_FOUND` — нет источника или fetch неуспешен.  
* `ERR_PROVIDER_UNAVAILABLE` — есть candidate, но нет доступного адреса/соединения.  
* `ERR_NODE_CID_MISMATCH` / `ERR_NODE_SIG_INVALID` — результаты верификации Node.
* `ERR_UFA_UNSUPPORTED` — строка не распознана как alias/ID.
* `ERR_UFA_TYPE_MISMATCH` — UFA не соответствует контексту клиента (например, service_id в Node Browser).
* `ERR_SERVICE_NAME_REQUIRED` — невозможно вычислить `service_id` без `service_name`.

Ошибки **ДОЛЖНЫ** быть классифицируемыми (`ERR_*`) в соответствии с SPEC-140.

---

## 10) Совместимость

* SPEC совместим с текущими wire-форматами.  
* Реализация **НЕ ДОЛЖНА** требовать изменения существующих задач, envelope или node форматов.
