# SPEC-100: Network Manager (NET)

**Статус:** Approved v1.0 (2026-02-02)  
**Зависимости:** SPEC-000, SPEC-001, SPEC-002, SPEC-010  
**Назначение:** определить состояние узла, запуск/останов сети, управление соединениями и лимитами ресурсов.

---

## 1) Scope / Out-of-scope

### Scope

* Состояния сети (on/off/degraded).
* Конфигурация bootstrap и локальных лимитов.
* Управление входящими/исходящими соединениями.
* Политики ретраев и backoff.

### Out-of-scope

* Глобальный DHT, автоматический поиск “всех” пиров.
* Сложные стратегии оптимизации маршрутизации.

---

## 2) Состояния NET

NET **ДОЛЖЕН** быть конечным автоматом со статусами:

* `stopped` — сеть выключена.
* `starting` — инициализация ключей/конфига/транспорта.
* `online` — сеть функционирует.
* `degraded` — сеть частично функционирует (нет bootstrap/мало соединений/ошибки транспорта).
* `stopping` — завершение соединений и сохранение состояния.

Переходы:

* `stopped -> starting -> online`
* `online -> degraded` при нарушении SLO соединений/ошибках транспорта
* `degraded -> online` при восстановлении
* `online|degraded -> stopping -> stopped`

`degraded` — рабочее состояние (не “фатальная ошибка”): узел продолжает обслуживать то, что может, и сообщает причины деградации через OBS (см. SPEC-420).

---

## 2.1 Причины деградации (фиксировано)

NET **ДОЛЖЕН** иметь набор причин деградации (могут накапливаться):

* `no_bootstrap`
* `low_peers`
* `transport_errors`
* `clock_skew`

---

## 3) Конфигурация v1 (фиксировано)

Файл конфигурации узла: `config/node.json` (UTF-8, JSON).

Минимальная схема:

* `node_name` (string, локально)
* `listen`:
  * `quic` (bool, default true)
  * `quic_addr` (string, default `0.0.0.0:0`)
* `bootstrap_peers` (array of объектов):
  * `peer_id`
  * `addrs` (array of string; формат addr см. SPEC-110)
* `limits`:
  * `max_inbound_conns` (uint)
  * `max_outbound_conns` (uint)
  * `max_msg_bytes` (uint)
  * `max_inflight_msgs` (uint)
  * `ban_window_ms` (int64)
* `pow`:
  * `default_difficulty` (uint)

Любые отсутствующие поля **ДОЛЖНЫ** заполняться значениями по умолчанию, зафиксированными в реализации.

---

## 4) Лимиты (обязательные)

NET **ДОЛЖЕН** enforce-ить:

* максимальное число одновременных соединений inbound/outbound;
* максимальный размер входящего сообщения (до декодирования payload);
* лимиты на попытки handshake с одного `peer_id`/адреса;
* бан-лист по `peer_id` с TTL = `ban_window_ms`.

---

## 4.1 Обработка проблем времени (clock skew) (фиксировано)

NET **ДОЛЖЕН** детектировать устойчивую проблему локального времени:

* если при handshake (см. SPEC-110) более чем с 3 различными peer подряд наблюдается `|remote_ts_ms - local_now_ms| > 5 минут`,
  * NET **ДОЛЖЕН** перейти в `degraded` с причиной `clock_skew`;
  * узел **ДОЛЖЕН** продолжать принимать входящие соединения, но **ДОЛЖЕН** отклонять сообщения с `ERR_HANDSHAKE_TIME_SKEW` / `ERR_TTL_EXPIRED` согласно правилам SPEC-110/140;
  * узел **ДОЛЖЕН** логировать предупреждение уровня `warn` с `event=net.clock_skew`.

Автоматическая “синхронизация времени” протоколом в v1 **НЕ ДОПУСКАЕТСЯ** (только действия оператора/ОС).

---

## 5) Ретраи (фиксировано)

Для исходящих соединений используется exponential backoff:

* base = 250ms
* factor = 2
* max = 30s
* jitter = ±20%

---

## 6) События NET (для наблюдаемости)

NET **ДОЛЖЕН** генерировать события:

* `net.started`, `net.stopped`
* `peer.connected`, `peer.disconnected`
* `peer.banned`, `peer.unbanned`
* `net.degraded`, `net.recovered`

Формат событий и логирование описаны в SPEC-420.
