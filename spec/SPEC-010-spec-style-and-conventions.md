# SPEC-010: Стиль спецификаций и конвенции

**Статус:** Approved v1.0 (2026-02-02)  
**Назначение:** единые правила языка, форматов, времени и сериализации для всех SPEC.

---

## 1) Нормативные ключевые слова

Ключевые слова **ДОЛЖЕН**, **НЕ ДОЛЖЕН**, **СЛЕДУЕТ**, **НЕ СЛЕДУЕТ**, **МОЖЕТ** интерпретируются как в RFC 2119.

Если формулировка допускает “выбор реализации”, это **ошибка спецификации**. В SPEC допускаются:

* выбор только при явной фиксации “НЕ ВХОДИТ В v1” (out-of-scope);
* расширения только через версионирование форматов/сообщений.

Если поведение не описано в SPEC, реализация **НЕ ДОЛЖНА** его “додумывать” как часть протокола (допускается только локальная внутренняя оптимизация, не влияющая на wire-поведение и доверенные инварианты).

---

## 2) Кодировки и представления

* Все документы SPEC **ДОЛЖНЫ** быть в UTF-8.
* Все строки в wire-форматах **ДОЛЖНЫ** быть в UTF-8.
* Текстовые идентификаторы **ДОЛЖНЫ** быть lowercase (если формат допускает регистр).
* Для идентификаторов v1 используются multiformats:
  * `identity_id`: `did:key` (multibase base58btc)
  * `peer_id` / `service_id` / `channel_id`: `multihash(sha2-256, bytes)` + `multibase(base32lower, multihash)`
  * `node_id`: CIDv1 (base32 lower)

---

## 3) Время

* Все времена **ДОЛЖНЫ** быть в UTC.
* Все метки времени в wire-форматах **ДОЛЖНЫ** быть `int64` unix time в миллисекундах (`*_at_ms`, `ts_ms`).
* Длительности/TTL **ДОЛЖНЫ** быть `int64` в миллисекундах (`ttl_ms`).

---

## 4) Криптография (базовые решения v1)

* Единственный подпись/ключи v1: **Ed25519**.
* Хэш-функция для идентификаторов/дедупликации v1: **SHA-256**.
* Для симметричного шифрования приватных payload (см. SPEC-220) используется **XChaCha20-Poly1305**.

Любые дополнительные алгоритмы **НЕ ВХОДЯТ В v1** и требуют отдельной SPEC-ветки версионирования.

---

## 5) Сериализация и каноникализация

### 5.1 Wire-формат сообщений сети

* Wire-формат overlay-сообщений **ДОЛЖЕН** быть CBOR (RFC 8949).
* Для подписей/хэшей CBOR **ДОЛЖЕН** использовать канонический режим (deterministic/canonical CBOR).
* Для v1 допускается опираться на детерминированность выбранной библиотеки при наличии golden tests для критичных структур.

### 5.2 Content Nodes

* Content Nodes **ДОЛЖНЫ** использовать CIDv1 + `dag-cbor` (см. SPEC-001 и SPEC-200).
* Любой объект, влияющий на `node_id`, **ДОЛЖЕН** иметь однозначную каноническую сериализацию.

---

## 6) Версионирование

* Любой wire-формат **ДОЛЖЕН** иметь поле `v` (целое), начинающееся с 1.
* Изменения, ломающие совместимость, **ДОЛЖНЫ** увеличивать `v`.
* Сервисные API **ДОЛЖНЫ** версионироваться в имени (`*.v<major>`) (см. SPEC-001).

---

## 7) Реализация (в рамках v1)

* Для canonical CBOR, dag-cbor, CID/multihash/multibase и криптографии **РАЗРЕШЕНО** использовать внешние библиотеки при условии, что:
  * они поддерживаются и не являются “заброшенными”;
  * режимы deterministic/canonical проверяемы тестами.
* Самописные реализации криптографии/кодеков **НЕ РЕКОМЕНДУЮТСЯ** для v1.
