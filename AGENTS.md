# AGENTS.md (root)

## Контекст проекта

Проект `ardents` — реализация распределённой overlay-сети для сервисов и контента. Нормативные требования к протоколу и форматам описаны в `spec/` (см. `spec/SPEC-000-system-overview.md`).

Язык реализации: **Go**. Архитектурная парадигма: **DDD + явное разделение слоёв** (domain / application / infrastructure / transport).

## Общие правила для изменений

* Любое изменение кода **ДОЛЖНО** соответствовать спецификациям в `spec/`. Если спецификация неполна — сначала меняем/добавляем SPEC.
* Никаких “вариантов на выбор” в документах: решения фиксируются явно (см. `spec/SPEC-010-spec-style-and-conventions.md`).
* Безопасность/стабильность важнее скорости разработки: предпочитаем простые и проверяемые решения, даже если они менее “быстрые”.

## Технические требования к коду

Нормативные требования к качеству, производительности, структуре и ограничениям на размер кода описаны в `docs/TECH-000-engineering-requirements.md`.

## Структура репозитория (ожидаемая)

* `cmd/peer/` — основной runtime узла (NET/MSG/Routing/Storage).
* `cmd/gateway/` — integration gateway (см. SPEC-320).
* `cmd/replay/` — утилита replay (см. SPEC-430).
* `cmd/sim/` — симулятор (см. SPEC-430).
* `internal/`
  * `internal/<bc>/domain/` — доменная модель (сущности/VO/инварианты/доменные события); **без** зависимостей от инфраструктуры.
  * `internal/<bc>/app/` — application layer (use-cases, orchestration, транзакционные границы).
  * `internal/<bc>/infra/` — реализации репозиториев, сети, хранилищ, крипто-адаптеров.
  * `internal/<bc>/transport/` — входные/выходные адаптеры (QUIC, HTTP gateway, CLI/TUI).
  * `internal/shared/` — общие типы/утилиты (строго без бизнес-логики).

Правило зависимостей:

* `domain` не импортирует `app/infra/transport`.
* `app` импортирует `domain`.
* `infra` и `transport` импортируют `app` и `domain`.

## Стандарты кода (Go)

* Форматирование: `gofmt` (обязательно), `goimports` (если используется).
* Стиль ошибок:
  * Ошибки **ДОЛЖНЫ** быть классифицируемыми (error code), особенно для протокольных отказов (см. SPEC-140).
  * Для наружных границ (transport) ошибки **ДОЛЖНЫ** маппиться в стабильные коды/статусы.
* Время: только UTC, `*_ms` (см. SPEC-010).
* Идентификаторы: строго по SPEC-001.
* Сериализация:
  * overlay messages: canonical CBOR (SPEC-010/140)
  * content nodes: CIDv1 + dag-cbor (SPEC-001/200)

## Тестирование (ожидаемое)

* Минимум: unit-тесты доменных инвариантов и кодеков (CBOR/CID/signatures).
* Контрактные тесты: валидация wire-форматов Envelope/Node против фиксаций (golden files).
* Интеграционные тесты: handshake + delivery + fetch в локальном окружении (N=2..3 peer).
